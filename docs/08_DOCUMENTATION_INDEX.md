### Repository-Wide Quality System

The reusable quality gates are maintained outside `docs/` in [Enterprise Quality Standard](../quality/QUALITY_STANDARD.md). The documentation-specific standard in this directory narrows that system to documentation governance and does not duplicate it.

## Relationships

- Foundation documents govern project direction, architecture, business rules, technology, and repository standards.
- Governance documents control document discovery, lifecycle, terminology, review, quality, decisions, and unresolved work.
- Delivery documents translate approved governing material into implementation and operational guidance.
- ADRs preserve individual decisions; the Decision Log indexes them.
- Topic folders contain supporting detail and must defer to their corresponding top-level governing document.
- Audit reports record findings and do not establish business or architecture decisions.
- Repository-intelligence documents describe collaboration, metadata, traceability, reading, and document relationships; they do not authorize product AI features or implementation.
- Repository Health and audit reports record current-state evidence; they do not establish governing authority.
- Git Governance defines proposed repository evolution rules and does not create a baseline, branch, commit, tag, remote, backup, or release by documentation alone.
- WordPress Enterprise Architecture defines proposed platform boundaries and records WordPress-specific Founder constraints; it does not install or configure the platform.
- Product-data documents define logical entities and mapping policies; they do not create products, taxonomies, attributes, inquiries, settings, schemas, imports, UI, or code.
- Information Architecture documents organize approved reference entities into logical site, URL, search, discovery, and linking structures; they do not create pages, menus, routes, indexes, links, plugins, configuration, or implementation.
- Content and Entity Architecture documents define logical content/entity/media/semantic responsibilities and relationships; they do not create content, platform objects, markup, media, SEO output, AI/LLM capabilities, plugins, configuration, or code.
- WordPress Solution Blueprint documents allocate future platform responsibilities and decision gates; they do not install, configure, register, build, select unresolved vendors, or authorize implementation.
- Release-engineering documents freeze and evaluate the current repository; only the exact local baseline is approved, while readiness, roadmap, and engineering guidance remain Review-state and do not authorize implementation.

## Dependency Graph

```text
Project Bible
  -> Project Constitution
      -> Business Rules
      -> Enterprise Architecture
          -> Technology Stack
              -> WordPress Architecture
                  -> Delivery, Security, SEO, UX, and Testing documents

Document Lifecycle + Document Template + Review Process + Quality Standard
  -> apply to every active document

Founder Decision Log + Open Questions
  -> feed approved decisions into Business Rules, Architecture, Technology, or ADRs

ADRs
  -> indexed by Decision Log
  -> referenced by affected governing and delivery documents

AI Collaboration + Metadata + Reading Order + Knowledge Graph
  -> improve repository interpretation and handoff
  -> do not authorize a Phase 1 AI product feature

Decision Classification + Lifecycle + Relationship Model + Rule Inheritance
  -> govern decision identity, document state, dependency direction, and conflict handling

Repository Health + Audit Reports
  -> record current-state validation evidence
  -> never become governing authority without explicit Founder approval

Repository Standards -> Git Governance
  -> Branches + Versions + Tags + Baselines + Commits + Releases + Backups
  -> no Git mutation without exact approval

Enterprise Architecture + Business Rules + Technology Stack + ADR 0001
  -> WordPress Enterprise Architecture
      -> future WordPress, WooCommerce, Blocksy, Elementor, content, inquiry,
         SEO, security, performance, and integration implementation evidence

WordPress Enterprise Architecture + Business Rules
  -> Enterprise Product Data Model
      -> Product Taxonomy Model + Product Attribute Model
      -> WooCommerce Product Model
          -> Inquiry Data Model
              -> future CRM/ERP/CentralSteel mappings

  -> Draft related context pending Founder disposition
  -> does not govern or block interpretation of the Review-state Product Data Model

WordPress Enterprise Architecture + Product/Taxonomy/Attribute/Inquiry Models
  -> Enterprise Information Architecture
      -> Site Structure + URL Architecture
          -> Search and Discovery + Internal Linking Model
              -> future configuration and validation evidence

Information Architecture + Product/Taxonomy/Attribute/Inquiry Models
  -> Enterprise Content Architecture
      -> Entity Relationship Model
          -> Schema.org Strategy + Content Types + Media Strategy + SEO Entity Model

WordPress Architecture + Product/IA/Content/Entity Models + ADR 0001
  -> Enterprise WordPress Solution Blueprint
      -> Blocksy + Elementor + WooCommerce Blueprints
      -> CPT + Taxonomy + Field Blueprints

Accepted principles + ADR 0001 + current repository state
  -> Repository Baseline v1.0 + Release Notes
      -> Implementation Readiness + Sprint Roadmap + Engineering Guidelines
          -> future separately approved implementation sprints
```
